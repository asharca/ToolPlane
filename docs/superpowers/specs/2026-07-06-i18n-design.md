# i18n 中英切换完整补全设计

**日期**: 2026-07-06  
**分支**: `feat/i18n`  
**范围**: 前端所有静态字段支持中英切换，MCP skill/server/client 的原始描述除外。

## 背景

项目已集成 `next-intl`（见 `package.json:36`、`next.config.ts`、`src/i18n/routing.ts`），并已有：
- `messages/en.json` / `messages/zh.json`：部分 namespace（`common`、`header`、`footer`、`home`、`search`、`server`、`auth`、`errors`、`metadata`、`console.*`）已翻译。
- `src/components/layout/LocaleSwitcher.tsx`：EN/中切换按钮。
- `src/lib/i18n/actions.ts`：`setLocale` Server Action，写入 `NEXT_LOCALE` cookie 并持久化到 `User.locale`。
- `tests/unit/i18n-completeness.test.ts`：强制 `zh.json` 与 `en.json` key 完全一致。

当前问题是仍有大量组件/页面使用硬编码英文（或零散中文），导致中文内容不完整。

## 目标

1. 所有前端静态文案（public 站点、admin 后台、认证页、控制台、安装页）都能随 `LocaleSwitcher` 中英切换。
2. 不翻译来自数据库的目录数据（MCP server/skill/client 原始标题和描述）。
3. 不引入新的 i18n 方案或重型依赖，复用现有 `next-intl` 结构。
4. 新增翻译 key 后，`i18n-completeness.test.ts` 必须继续通过。

## 排除项

- MCP skill / server / client 原始标题、描述（来自 Prisma 目录表）。
- 代码技术常量、className、URL slug、日志、错误堆栈。
- 用户自定义输入（workspace 名、token 名、agent 名等）。

## 方法：AST 候选提取 + 人工确认替换

由于项目使用 `next-intl` 而非 `i18next`，`i18next-parser` 不能直接把硬编码字符串转成 `t()` 调用。因此采用一次性脚本扫描 + 人工 review 的方式：

1. 新增 `scripts/extract-i18n-candidates.ts`，使用 TypeScript Compiler API 扫描：
   - `src/app/(site)/**`
   - `src/app/admin/**`
   - `src/app/app/(auth)/**`
   - `src/app/app/[workspace]/**`
   - `src/app/install/**`
   - `src/components/**`
2. 脚本输出候选清单（JSON），包含文件路径、行号、原始文案、建议 namespace。
3. 人工 review 清单，把需要翻译的条目写入 `messages/en.json` 和 `zh.json`。
4. 在源码中把确认过的字符串替换为 `useTranslations` / `getTranslations`。

脚本不直接改写源文件，避免误伤动态数据或技术常量。

## Key 命名与 namespace

沿用并扩展现有结构：

| Namespace | 用途 |
|-----------|------|
| `common.*` | 通用按钮、状态、占位文案 |
| `header.*` / `footer.*` | 站点导航 |
| `home.*` / `search.*` / `server.*` | public 页面 |
| `auth.*` | 登录/注册 |
| `console.sidebar.*` | 控制台侧边栏 |
| `console.agents.*` / `console.mcp.*` / `console.skills.*` / `console.toolkits.*` / `console.observability.*` / `console.members.*` / `console.settings.*` | 控制台各页面 |
| `errors.*` | 404/500/未授权页面 |
| `metadata.*` | 页面 title / description |

新增 key 使用 camelCase，按页面/组件分层，避免扁平化。

## 实施步骤

1. `git checkout -b feat/i18n`（已完成）。
2. 编写并运行 `scripts/extract-i18n-candidates.ts`，生成候选清单。
3. 人工 review 清单，分批写入 `messages/en.json` 和 `messages/zh.json`。
4. 替换页面/组件中的硬编码字符串：
   - Client Components：`const t = useTranslations('namespace')`。
   - Server Components：`const t = await getTranslations('namespace')`。
   - `metadata` 使用 `generateMetadata({ params })` 按 locale 返回。
5. 在 `DashboardHeader` 或 `DashboardSidebar` 中补充 `LocaleSwitcher`，使控制台也能切换语言（若当前只在站点 Header 中）。
6. 运行测试：
   - `pnpm test`（重点 `i18n-completeness.test.ts`）
   - `pnpm lint`
   - `pnpm build`
7. 视觉抽检：切换 EN/中，确认无漏译、无布局崩坏。

## 测试策略

- `tests/unit/i18n-completeness.test.ts`：强制 `zh.json` 与 `en.json` key 完全一致，作为翻译完整性闸门。
- `pnpm lint` / `pnpm build`：保证类型和编译通过。
- 浏览器抽检：验证切换器在站点和控制台均生效，中文模式无英文残留。

## 风险与兜底

- AST 脚本可能漏掉模板字符串或 JSX 表达式中的文案，人工 review 兜底。
- 脚本可能误判技术常量，review 时排除。
- 如果运行中发现自动化收益低于预期，可随时切换为逐文件手工替换；key 结构和测试策略不变。
